# Задание для ИИ-агента: исправление, верификация и оптимизация `broken-app`

## Цель

У вас есть два проекта:

- `broken-app` — проект с дефектами, неоптимальными участками и потенциально опасным кодом, который нужно исправить и ускорить.
- `reference-app` — эталонная реализация для сверки поведения.

Цель работы:

- найти и исправить дефекты в `broken-app`;
- подтвердить отсутствие UB (undefined behavior);
- устранить утечки и ошибки памяти;
- проверить потокобезопасность;
- оптимизировать критичные участки;
- подтвердить ускорение измерениями;
- подготовить полный набор артефактов для ревью.

## Что заранее подготовлено в `broken-app`

В проекте уже заложены:

- минимум **5 разных дефектов**, включая:
  - `off-by-one`;
  - `use-after-free` / aliasing в `unsafe`;
  - утечку памяти;
  - неверную логику;
  - низкоэффективный алгоритм;
  - гонку данных в модуле с потоками;
- падающие тесты на часть ошибок;
- комментарии-метки, помогающие найти остальные проблемы;
- намеренно медленные функции:
  - с лишними аллокациями;
  - с плохой асимптотикой;
- простые бенчмарки «до»:
  - `criterion`, либо
  - `harness = false`.

## Роль ИИ-агента

ИИ-агент должен действовать как инженер по отладке, верификации и производительности. Его задача — не просто «починить тесты», а выполнить полноценный технический цикл:

1. воспроизвести проблемы;
2. локализовать и исправить дефекты;
3. доказать корректность исправлений;
4. найти узкие места;
5. улучшить производительность;
6. собрать воспроизводимые артефакты.

---

## Правила работы

При выполнении задания придерживайтесь следующих принципов:

- не менять поведение системы произвольно — ориентироваться на `reference-app`;
- любые исправления подтверждать тестами;
- любые оптимизации подтверждать замерами;
- не оставлять `unsafe`, если его можно убрать без потери смысла;
- если `unsafe` сохраняется, он должен быть локализован, обоснован и проверен;
- избегать «магии», неочевидных хаков и лишних зависимостей;
- предпочитать стандартную библиотеку, простые структуры данных и читаемые решения;
- не ухудшать воспроизводимость результатов.

## Пошаговый план выполнения

## Шаг 1. Ознакомление и первичная фиксация состояния

### Что нужно сделать

1. Клонировать оба проекта:
   - `broken-app`
   - `reference-app`

2. В `broken-app` выполнить:
   - `cargo check`
   - `cargo test`
   - при наличии — `cargo run --bin demo`

3. Зафиксировать:
   - ошибки компиляции;
   - падающие тесты;
   - сообщения panic;
   - подозрительные места в `unsafe`;
   - участки с потоками;
   - функции с очевидно плохой производительностью.

4. В `reference-app`:
   - собрать проект;
   - прогнать тесты;
   - при необходимости запустить демонстрационные сценарии;
   - зафиксировать ожидаемое поведение, формат вывода и семантику.

### Результат шага

Должно быть понятно:

- какие баги уже проявляются явно;
- какое поведение считается корректным;
- какие части требуют сверки с эталоном.

---

## Шаг 2. Поиск и исправление дефектов

### Обязательные инструменты

Используйте следующие инструменты:

- отладчик:
  - `gdb` или `lldb`;
- Miri:
  - `cargo +nightly miri test`;
- Valgrind:
  - `valgrind --leak-check=full cargo test --tests`;
- sanitizer’ы на nightly:
  - AddressSanitizer;
  - ThreadSanitizer.

### Рекомендуемый порядок

1. Воспроизвести логические ошибки под отладчиком.
2. Проверить код через Miri и найти UB.
3. Проверить проект через Valgrind:
   - утечки;
   - invalid read/write;
   - use-after-free;
   - double free.
4. Прогнать ключевые тесты с:
   - `RUSTFLAGS="-Zsanitizer=address"`
   - `RUSTFLAGS="-Zsanitizer=thread"`
5. Исправить все найденные проблемы.
6. По возможности:
   - удалить `unsafe`;
   - либо сузить его область;
   - либо заменить безопасной альтернативой.

### Требования к исправлениям

Для каждого найденного дефекта нужно:

- описать проблему;
- указать, как она воспроизводилась;
- указать, чем подтверждалась;
- внести исправление;
- добавить регрессионный тест.

### Минимальный набор категорий дефектов, который должен быть закрыт

- ошибка границ (`off-by-one`);
- ошибка памяти / UB;
- утечка;
- логическая ошибка;
- неэффективный алгоритм;
- гонка данных / потоковая проблема.

---

## Шаг 3. Подтверждение корректности

После исправлений нужно повторно прогнать верификацию.

### Обязательные проверки

- `cargo test`
- `cargo +nightly miri test`
- `valgrind --leak-check=full cargo test --tests`
- ключевые тесты с ASan
- ключевые тесты с TSan

### Дополнительно

На каждый найденный баг должен быть добавлен отдельный регрессионный тест или понятный тестовый сценарий.

### Критерий завершения шага

- тесты проходят;
- Miri не сообщает об UB;
- Valgrind не сообщает об утечках и invalid memory access;
- sanitizer’ы не находят проблем на целевых тестах.

---

## Шаг 4. Поиск узких мест

### Что нужно сделать

Провести профилирование сценариев из:

- `src/bin/demo.rs`,
- интеграционных тестов,
- дополнительных сценариев, если они лучше нагружают горячие участки.

### Инструменты

Предпочтительно использовать:

- `perf`;
- `flamegraph`;
- при необходимости:
  - `heaptrack`,
  - `massif`.

### Что нужно зафиксировать

- какие функции занимают основное время;
- где происходят лишние аллокации;
- какие участки имеют плохую асимптотику;
- какие пути действительно являются горячими.

### Артефакты

Сохранить:

- flamegraph в PNG/SVG;
- текстовое описание узких мест;
- при наличии — вывод `perf stat` / `perf report`.

---

## Шаг 5. Бенчмарки до оптимизации

### Требования

Нужно подготовить воспроизводимые бенчмарки в `benches/`.

Рекомендуется:

- `criterion`;
- либо совместимый текстовый harness.

### Бенчмарки должны покрывать

- горячие пути;
- сценарии с «большим» входом;
- те функции, которые реально попали в профиль;
- места, где ожидается эффект от оптимизации.

### Что нужно сохранить

- результаты «до»;
- конфигурацию входных данных;
- команду запуска;
- файлы отчётов в `artifacts/`.

Примеры имён:

- `artifacts/baseline_before.txt`
- `artifacts/criterion_before/`
- `artifacts/perf_before.txt`

---

## Шаг 6. Оптимизация

### Минимальные требования

Сделать **минимум две оптимизации**:

1. **Алгоритмическая**
   - улучшение асимптотики;
   - замена структуры данных;
   - устранение квадратичности;
   - предвычисление, индексация, буферизация и т.п.

2. **Микрооптимизация**
   - уменьшение числа аллокаций;
   - сокращение копирований;
   - отказ от лишних `clone`;
   - оптимизация проходов по данным;
   - работа со срезами, буферами, итераторами.

### Допустимые направления

- убрать лишние промежуточные `Vec` / `String`;
- заменить повторный поиск на индекс / map / set;
- сократить количество проходов по коллекциям;
- использовать `with_capacity`, где это обосновано;
- заменить неудачный алгоритм на более подходящий;
- устранить повторные преобразования форматов;
- снизить contention в многопоточном коде, если это оправдано.

### Ограничения

- не добавлять лишние зависимости без необходимости;
- не делать код менее читаемым ради микроскопического выигрыша;
- не применять микрооптимизацию до подтверждения hot path;
- не ломать соответствие `reference-app`.

---

## Шаг 7. Проверка после оптимизации

После оптимизаций необходимо заново выполнить полный цикл проверок.

### Обязательный повторный прогон

- `cargo test`
- `cargo +nightly miri test`
- `valgrind --leak-check=full cargo test --tests`
- ASan
- TSan
- `cargo bench`

### Что нужно сравнить

- время выполнения до/после;
- число аллокаций, если измерялось;
- при наличии — инструкции / cache misses / CPU cycles;
- стабильность результатов на одинаковых входах.

### Что нужно задокументировать

- какие функции ускорены;
- за счёт чего достигнут выигрыш;
- насколько ускорение воспроизводимо;
- нет ли регрессий в корректности.

## На что будет смотреть ревью

При проверке особое внимание уделяется следующему:

### Корректность инженерного процесса
- использовались ли реально:
  - отладчик,
  - Miri,
  - Valgrind,
  - sanitizer’ы;
- есть ли подтверждение прогонов.

### Качество фиксации «до» и «после»
- есть ли численные результаты;
- сравниваются ли одинаковые входы;
- есть ли понятные артефакты.

### Обоснованность оптимизаций
- есть ли профиль до оптимизации;
- улучшалась ли действительно горячая часть;
- есть ли алгоритмические улучшения, а не только косметика.

### Безопасность и надёжность
- не появился ли новый UB;
- не остались ли утечки;
- не сломалась ли потоковая безопасность.

### Качество тестирования
- есть ли регрессионные тесты;
- проверяют ли они именно зафиксированные ошибки;
- воспроизводимы ли бенчмарки и тесты.

## Как сдавать проект

После завершения работы итоговый репозиторий должен находиться в публичном GitHub-аккаунте.

### Порядок сдачи

1. Подготовить репозиторий:
   - сложить все артефакты в папку проекта;
   - убедиться, что все файлы закоммичены;
   - проверить сборку:
     ```bash
     cargo build --workspace
     ```
   - проверить README.

2. В тренажёре:
   - вставить ссылку на GitHub-репозиторий в поле для ссылки;
   - нажать «Отправить».

   Пример:
   ```text
   https://github.com/your-username/blog-project
   ```

3. Дождаться проверки:
   - ревьюер проверит проект;
   - запустит компоненты;
   - посмотрит код и артефакты.

4. Если будут замечания:
   - ревьюер создаст комментарии / Issues на GitHub;
   - нужно внимательно прочитать замечания;
   - при необходимости задать уточняющие вопросы в комментариях к Issue.

5. Исправление замечаний:
   - внести изменения;
   - закоммитить исправления;
   - ответить в Issue, что именно исправлено.

6. Повторная проверка:
   - ревьюер перепроверит проект;
   - если замечания останутся, появятся новые комментарии;
   - если всё закрыто — проект принят.

7. Финальный статус:
   - после закрытия всех Issues появится сообщение о принятии проектной работы.

## Инструкция ИИ-агенту по стилю выполнения

При выполнении задания агент должен:

- работать поэтапно;
- после каждого крупного этапа фиксировать результат;
- не смешивать исправление багов и оптимизацию без замеров;
- не считать задачу завершённой только потому, что тесты перестали падать;
- отдельно проверять memory safety;
- отдельно проверять concurrency safety;
- отдельно подтверждать performance improvement;
- оформлять результаты так, чтобы ревьюер мог всё воспроизвести.

---

## Критерий готовности

Работа считается завершённой, если одновременно выполнены все условия:

- `broken-app` ведёт себя корректно относительно `reference-app`;
- все известные дефекты устранены;
- добавлены регрессионные тесты;
- UB не обнаруживается;
- утечек и memory errors нет;
- data race на ключевых сценариях не воспроизводится;
- выполнены и сохранены бенчмарки до/после;
- есть хотя бы одна обоснованная алгоритмическая оптимизация;
- есть хотя бы одна обоснованная микрооптимизация;
- ускорение подтверждено артефактами;
- репозиторий готов к внешнему ревью.

---

## Короткая исполнительная формулировка для агента

Исправьте `broken-app`, используя `reference-app` как эталон поведения. Найдите и устраните все существенные дефекты, включая UB, утечки, логические ошибки и проблемы потоков. Подтвердите корректность с помощью тестов, Miri, Valgrind и sanitizer’ов. Затем найдите узкие места, зафиксируйте baseline, выполните минимум одну алгоритмическую и одну микрооптимизацию, повторно подтвердите корректность и задокументируйте ускорение. Подготовьте воспроизводимые артефакты и репозиторий, готовый к публичному ревью.
